# 主编协作手册 · 用户视角

> **给用户（你）看的**。讲清楚：主编这一级是干什么的、你（用户）和主编怎么配合、一章 / 一季的生命周期、什么时候该决策、出了事怎么办。
> 配套：`docs/standards/CLAUDE.md`（执笔规则）· `.claude/skills/zhubian/SKILL.md`（主编 skill）· `.claude/skills/bianjibu/SKILL.md`（编辑部 skill）· `.claude/skills/wenbi-review/SKILL.md`（审校 skill）

---

## §〇 这套协作是干什么的

open-souls 是一个**多季、多魂、群像无限流**的网文生产系统。规模大了之后，单个 subagent 装不下整个流水线，所以我们分了**三层角色**：

```
用户（你）
   ↓ 决策 + 验收
主编（zhubian）· 拆活 / 派活 / 收活 / 验收 / 周报
   ↓ 派活 / 收活
编辑部（bianjibu）· 内容生产 / 骨架 / 章节正文
审校部（wenbi-review）· 文笔审阅 / 质量底线
```

**这套协作的目标**：

1. **让你（用户）只做决策，不做执行。** 你的时间应该花在「这季主不主流」「这个魂 carry 对不对」「这章过了没有」，而不是「这句话怎么改」。
2. **让流水线可调度、可回滚、可扩展。** 哪一节卡了，主编单独派活补救；不会因为一个 subagent 出问题整个停摆。
3. **让质量分层。** 编辑部负责"是不是想要的"，审校部负责"够不够好"，主编负责"是不是按计划推的"——三个角色不互相越界。

## §一 三个角色的边界与责任

| 角色 | 干什么 | 不干什么 | 你（用户）找谁 |
|---|---|---|---|
| **主编**（zhubian） | 拆活、派活、收活、验收、写周报、问你要决策 | 写正文、审文笔、替你做主题材决定 | 问进度 / 改方向 / 等周报 |
| **编辑部**（bianjibu） | 写 world.md / 章节正文 / 季度规划 / carry_memory 对齐 | 调其他 subagent、决定"过不过" | 改文 / 看 draft |
| **审校部**（wenbi-review） | 审文笔 / 校验双层文本 / 卡质量底线 | 调其他 subagent、决定"写什么" | 看审校意见 / 接受打回 |
| **你**（用户） | 拍板 / 验收最终产物 / 改方向 | 写正文、写文档、改字段（除非明确要你写） | — |

**判据**：

- 你想问"这章能不能过" → 看 wenbi-review 的审校意见，不是问主编。
- 你想问"季 02 写不写" → 问主编，主编会派 bianjibu 出候选方案。
- 你想问"这章写得对不对主角人设" → 问主编，主编会派 bianjibu 对照 soul.md 检查。

```
        │   · 主编在 dispatch.log 记录"完成"    │

```

### 一章通常耗时

- 接活 + 拆活：5 分钟
- 派活 + subagent 工作：30–45 分钟
- 收活 + 验收 + 合并：10 分钟
- 你的决策：1–10 分钟（取决于你是否要改）
- 入档 + 刷索引：5 分钟
- **总耗时：约 1 小时**

如果你（用户）的决策在 5 分钟内能拍板（"过"），全章可在 1.5 小时内闭环。

### 单章的"硬卡点"

- 你（用户）必须拍板的节点：**只有 §二 ⑥**
- 主编必须拍板的节点：**§二 ④（过/打回）**
- bianjibu 必须拍的节点：**文笔（与 wenbi-review 互动）**
- wenbi-review 必须拍的节点：**双层文本 + 七维底线**

## §三 一季的生命周期

### 一季的长度（按当前 config.yaml）

- 总章节：**约 400 回**（按季 01 节奏外推；季 02 可调）
- 节奏：每 6 小时一拍，每天约 4 回，约 **100 天一季**
- 四拍：起（≈100 回）/ 承（≈100 回）/ 转（≈150 回）/ 合（≈50 回）

### 一季的 5 个阶段

```
开季准备（1-2 周）
  ├── 主编派：bianjibu 出 5 题材候选 + world.md 骨架 + carry_memory 对齐
  ├── 主编派：wenbi-review 给文笔底线 review
  └── 你（用户）：拍板题材 / 拍板母题 / 拍板 active_tropes

开写（每 6 小时一拍）
  ├── 按 §二 单章流程循环
  ├── 主编每周一次周报
  └── 你（用户）：看周报 + 拍板偶尔的关键决策

中段（约 ch150）
  ├── 主编提一次"中段评估"——节奏是否按计划 / 是否需要补哪些角色
  └── 你（用户）：看评估 + 拍板是否调整

收尾（约 ch350 起）
  ├── 主编派：bianjibu 出"收尾方案"（决战在第几回 / 留几个钩接下季）
  └── 你（用户）：拍板收尾方式

收尾 + 开下季准备（约 ch400-500）
  ├── 收尾：决战揭半个真相 + 留钩
  ├── 同时：开下季准备（参考 ch502 收尾时开季 02 准备）
  └── 你（用户）：拍板下季题材 + 主基调
```

### 什么时候开下一季？

**触发条件**（任一即开）：
1. 决战回已发（按计划合上钩）
2. 你（用户）说"够了"
3. 连续 50 回节奏跑偏（编辑部和审校部都建议收）

**不要开新季的信号**：
- 上一季还没合钩
- 上季的 carry_memory 还有没收的线
- 你（用户）还没拍板下季题材

## §四 派活协议（主编怎么问你要决策，你（用户）怎么回复）

### 主编会问你什么？

主编**只在三件事**上问你：

1. **拍板**（选项已给，你勾一个）：题材、母题、active_tropes、收尾方式、下季方向。
2. **冲突 / 风险**（解释背景，给推荐）：subagent 出错、产物不一致、节奏严重跑偏。
3. **进度同步**（周报里）：本周推进 / 下周计划 / 需要决策的清单。

### 你（用户）怎么回复？

**回复格式（推荐）**：

```
决策项：[原文]
我的决定：[选项 A / 选项 B / 自定义]
理由（可选）：[1-2 句]
下一步：[让主编继续推进 / 改方向 / 暂停]
```

**示例**：

> 决策项：季 02 题材
> 我的决定：候选 E 都市悬疑刑侦
> 理由：可出片 + 19 魂 carry 损耗最小 + 男女频天然平衡
> 下一步：让 bianjibu 把 world-template 落到正式路径，准备开写

### 你（用户）不该回复什么？

- 不要给具体怎么写（"这章应该用 X 桥段"）——这是 bianjibu 的活。
- 不要改文笔（"这句不通顺"）——这是 wenbi-review 的活。
- 不要绕过主编直接派活给 subagent——主编会丢上下文。

### 决策的"门槛"

不是每个细节都要问你。**只在以下情形问**：

- 主题材 / 主基调 / 主角灵魂 carry 方式
- 季度级节奏调整（"季 02 比季 01 慢一倍"）
- subagent 卡住超过 60 分钟
- 既写章节需要回改（改情节层，不只是改文笔）
- 文档结构变动（如加新 skill、新目录）

**不问你**：

- 单章字数微调（在 320 字 ±20% 内）
- 单角色行为细节（除非影响剧情走向）
- 单桥段替换（playbook 库内）

## §五 协作节奏（什么时候并行，什么时候串行）

### 并行的情形（同时跑多个 subagent）

- 写新章 + 审上一章（不同章节，不冲突）
- 写章节正文 + 写章节索引（不同文件，不冲突）
- 出题材候选 + 出 world 骨架（不同文件，但都依赖 ties.json — 可以并行，写完对照）
- 写新季骨架 + 收尾上季决战（不同文件，完全不冲突）

### 串行的情形（必须先后）

- 先出 world.md 骨架 → 才能开写第一章
- 先出第 N 章正文 → 才能审第 N 章
- 先拍板题材 → 才能写 world-template（但评估草案可以先出）
- 先对齐 carry_memory → 才能写新季涉及该魂的章节

### 节奏控制（主编守的硬规则）

- **任何 slot 闲过 30 分钟 → 主编必须立即派活**（不让 subagent 资源空转）。
- **任何工单超 60 分钟无进展 → 主编必须处置**（见 §六）。
- **任何决策等待用户超过 24 小时 → 主编必须提醒**（周报里 + 单独 ping）。

### 你（用户）的节奏责任

- **决策别拖过 24 小时**。卡住的是你这一端，主编 / 编辑部 / 审校部都干不了。
- **周报别只看，必须回应**（哪怕只是"看到了，按计划推"）。
- **临时改方向要给缓冲**（"现在停下" → 主编需要 10-15 分钟处置正在跑的工单）。

## §六 失败案例与处理

### 情形 1 · subagent 卡住（无产物）

**主编动作**：
- 15 分钟无进展 → poll 一次
- 30 分钟无进展 → 杀 + 重派，附"接着上次做"
- 60 分钟无进展 → 视同失败，回滚到拆活

**你（用户）的动作**：等主编的周报或即时汇报；不要直接重派（主编会丢上下文）。

### 情形 2 · 烂尾章（写了一半 / 方向跑偏）

**主编动作**：
- 打回原文，让 bianjibu 续写 / 重写
- 如果打回 2 次还不行 → 换派活模板的"上下文"段，重派
- 如果换模板后还不行 → 升级到主编自己接手（极其罕见）

**你（用户）的动作**：看主编给你的"烂尾原因 + 处置方案"，勾个"同意处置"即可。

### 情形 3 · 文档不一致（两份文件字段冲突）

**主编动作**：
- 以季 01 既定字段为准拍板
- 通知相关 subagent 修正
- 在产物"备注"段记录这次冲突的处理依据

**你（用户）的动作**：通常不需要介入，除非冲突影响了你拍板的依据。

### 情形 4 · subagent 出现幻觉（编造人物 / 编造文档）

**主编动作**：
- 立刻停掉该 subagent
- 检查产物，剔除编造部分
- 复核派活模板是否给了足够上下文——是不是没给权威文件清单
- **绝不**让幻觉产物进入正式路径（哪怕临时路径也不行）

**你（用户）的动作**：在周报里看一下，避免类似幻觉再发生（如果反复出现，可能要换 subagent 模型）。

### 情形 5 · 主题材失败（季 02 写到一半发现题材选错了）

**主编动作**：
- 立刻冻结该季新章节派发
- 派 bianjibu 出"止损方案"：是改方向 / 收尾换题材 / 整体推倒重来
- 给你（用户）三选一：止损 / 收尾换 / 推倒

**你（用户）的动作**：这是季度级决策，必须你拍。主编会推荐一个，但**这是你的判断**。

### 情形 6 · 用户方向反复（"刚说做 A，又改 B"）

**主编动作**：
- 不抗议，但每次都明确告诉你"如果改 B，之前 A 的工单会损失多少"
- 必要时把反复改动记到周报里——让你（用户）看到自己的反复

**你（用户）的动作**：方向调整前问自己"这是真的判断，还是临时起意"——如果是后者，先 sleep 一晚再决定。

### 情形 7 · 主编本身出错（罕见）

如果主编派活失误 / 漏活 / 收活错：
- 你（用户）直接指出"这章没按 §二 流程"
- 主编立即自查 + 致歉 + 重走流程

## §七 一周一次的主编周报模板

主编每周日 20:00 给你（用户）发一份周报，格式如下：

```markdown
# open-souls · 主编周报 · YYYY-MM-DD

## 本周推进
- [已完成项 1]
- [已完成项 2]
- [已完成项 3]

## 当前状态
- 季 01：[ch###–ch###] · 进度 [X/Y] · 节奏 [正常/超前/落后]
- 季 02：[状态：未启动/候选中/已拍板/已开写]
- 文档：共 N 份 · 本周新增 M 份

## 本周卡住 / 风险
- [风险 1 + 处置]
- [风险 2 + 处置]

## 下周计划
- [周一-周三]
- [周四-周五]
- [周末]

## 需要用户决策
- [决策项 1：选项 A / 选项 B / 推荐]
- [决策项 2：选项 A / 选项 B / 推荐]

## 主编备注
- [本周观察到的任何非任务相关事项]
```

### 你（用户）看周报要做的事

- ✅ 看"本周推进"——确认进展符合预期
- ✅ 看"当前状态"——节奏没跑偏
- ✅ 看"卡住 / 风险"——必要时介入
- ✅ 看"需要用户决策"——勾选项，给下一步
- ❌ 不要逐字读"主编备注"——除非你想了解细节

### 周报回复模板

```
# 回复 · 周报 · YYYY-MM-DD

决策项 1：[勾选项]
决策项 2：[勾选项]
下周：[同意计划 / 调整某项]
```
